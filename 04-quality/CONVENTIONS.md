# CONVENTIONS

## Назначение

Этот документ фиксирует правила, которые делают проект единообразным: код, документация, имена, коммиты, структура и рабочие договоренности.

## Репозиторий

| Путь | Назначение | Владелец |
| --- | --- | --- |
| `/docs` | Документация проекта | Documentation / Tech Lead |
|  |  |  |

## Именование

| Объект | Правило | Пример |
| --- | --- | --- |
| Ветки | `<type>/<short-task>` | `feature/auth-flow` |
| Коммиты | Conventional Commits | `fix: handle empty state` |
| Документы | `UPPER_SNAKE_CASE.md` | `REVIEW_POLICY.md` |

## Код

- Использовать стиль, принятый в существующем проекте.
- Не смешивать unrelated refactor с функциональной задачей.
- Не добавлять зависимость без понятного сценария использования.
- Ошибки должны быть диагностируемыми.
- Публичные контракты должны быть задокументированы.

## Документация

- Один факт должен иметь один основной документ-владелец.
- Документы должны обновляться вместе с изменением поведения.
- Специфичные или временные материалы не должны разрастаться в альтернативную документацию.
- Устаревший документ нужно обновить, переместить в архив или удалить.

## Коммиты и PR

Рекомендуемые типы:

- `feat`
- `fix`
- `docs`
- `refactor`
- `test`
- `chore`
- `build`
- `ci`

## Проверки перед сдачей

- [ ] Изменение решает заявленную задачу.
- [ ] Нет несвязанных изменений.
- [ ] Пройдены релевантные тесты.
- [ ] Обновлена документация.
- [ ] Известные риски явно указаны.
