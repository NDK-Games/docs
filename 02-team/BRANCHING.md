# BRANCHING

## Назначение

Этот документ описывает правила веток. Ветки используются при необходимости: для параллельной работы, review, рискованных изменений или релизной изоляции.

## Когда отдельная ветка нужна

- Над задачей работают параллельно с другими изменениями.
- Изменение требует code review.
- Есть риск конфликтов или отката.
- Задача затрагивает релиз, инфраструктуру, миграции или безопасность.
- Работа займет больше одного короткого цикла.

## Когда можно без отдельной ветки

- Проект ведет один человек.
- Изменение маленькое и низкорисковое.
- Нет параллельной работы.
- Нет требования PR/review.

## Формат веток

| Тип | Формат | Пример |
| --- | --- | --- |
| Feature | `feature/<task>` | `feature/user-profile` |
| Fix | `fix/<bug>` | `fix/login-empty-state` |
| Docs | `docs/<topic>` | `docs/team-template` |
| Refactor | `refactor/<area>` | `refactor/auth-service` |
| Release | `release/<version>` | `release/2026-05-13` |
| Agent | `agent/<role>/<task>` | `agent/qa/regression-pass` |

## Правила

- Одна ветка - один связный scope.
- Не смешивать feature, refactor и unrelated cleanup.
- Держать ветки короткоживущими.
- Перед merge обновить ветку от целевой ветки, если есть конфликтующие изменения.
- После merge удалить ветку, если она больше не нужна.

## Merge policy

| Риск изменения | Требование |
| --- | --- |
| Низкий | Self-check или легкое ревью |
| Средний | Code review + релевантные тесты |
| Высокий | Tech Lead review + QA / Security при необходимости |

## Конфликты

Если две ветки меняют одну область:

1. Назначить одного owner конфликта.
2. Согласовать порядок merge.
3. Не переписывать чужие изменения без согласования.
4. После разрешения конфликта прогнать релевантные проверки.
