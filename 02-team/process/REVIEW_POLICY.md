# REVIEW_POLICY

## Назначение

Этот документ задает правила ревью для кода, документации, требований и релизов.

## Когда ревью обязательно

- Изменение затрагивает production-поведение.
- Изменение влияет на данные, безопасность, платежи, доступы или деплой.
- Есть архитектурное решение или новая зависимость.
- Работают несколько людей или агентов параллельно.
- Задача P0/P1.

## Виды ревью

| Вид | Кто делает | Что проверяет |
| --- | --- | --- |
| Product review | Product Manager / CEO | Ценность, scope, требования |
| Technical review | Tech Lead / CTO | Архитектура, поддерживаемость, риски |
| Code review | Developer Code Reviewer | Баги, регрессии, тесты |
| QA review | QA Agent | Сценарии, acceptance, regression |
| Security review | Security Agent | Секреты, доступы, угрозы |
| Docs review | Documentation Agent | Актуальность и отсутствие дублей |

## Требования к review

- Findings должны быть конкретными и ссылаться на файл, задачу или сценарий.
- Блокирующие замечания отделяются от пожеланий.
- Reviewer не должен требовать unrelated refactor.
- Если риск принят сознательно, это фиксируется в задаче или документе.

## Решения ревью

| Решение | Значение |
| --- | --- |
| Approve | Можно принимать |
| Approve with notes | Можно принимать, follow-up отдельно |
| Request changes | Нужны исправления до merge |
| Blocked | Не хватает информации или проверки |

## Checklist reviewer

- [ ] Scope соответствует задаче.
- [ ] Нет очевидных регрессий.
- [ ] Ошибки и edge cases обработаны.
- [ ] Тесты или ручная проверка достаточны для риска.
- [ ] Документация обновлена.
- [ ] Нет секретов и случайных данных.
